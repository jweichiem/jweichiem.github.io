Engineering

This page shows how the site is structured, what tradeoffs were made, and what quality standards guide the implementation.

The goal is not to present the most complex stack possible, but to demonstrate explicit architecture, maintainable boundaries, and thoughtful decisions.

What this site demonstrates

Explicit architecture

The project is small, but it is intentionally structured like production code instead of a one-off portfolio.

  • Shared route definitions drive navigation, metadata, and prerender behavior.
  • Client, entry, and server responsibilities are separated rather than mixed together.
  • Localization is handled as part of the architecture, not as an afterthought.

A small but complete stack

The stack stays compact while still supporting multiple deployment styles and a maintainable content model.

  • React, TypeScript, and SCSS provide a predictable frontend foundation.
  • SSR and SSG are both supported because deployment constraints differ.
  • The implementation favors control and clarity over framework-heavy abstraction.

System overview

One route model, several outputs

The route registry acts as a shared source of truth across the application.

  • Navigation labels come from the route model.
  • Route metadata is generated from the same source.
  • Prerender behavior is defined alongside the route itself.

Localized routing

Canonical English routes and Swedish path-prefixed routes are generated consistently.

  • Localized URLs stay readable.
  • Locale switching preserves path intent.
  • The same locale helpers are shared across client and server code.

Rendering strategy

The project supports both a Node SSR path and static prerendered output.

  • SSR is useful for demonstrating app architecture and server rendering.
  • Static output keeps deployment simple for GitHub Pages.
  • The rendering approach is flexible without making the project framework-dependent.

Key decisions and tradeoffs

Why support both SSR and SSG

The site is intentionally designed to show both runtime and static deployment options.

  • This adds some complexity, but it demonstrates portability across environments.
  • For a content-only site, pure static output could be enough.
  • I would revisit the dual path if the site stayed permanently static.

Why keep routing lightweight

A small routing layer makes the flow easier to understand and easier to control.

  • The route registry stays explicit.
  • Localized paths remain easier to reason about.
  • The tradeoff is less built-in convention than a larger framework would provide.

Why this remains intentionally small

The site is meant to demonstrate judgment, not to maximize implementation size.

  • No CMS is used because the content surface is still manageable in code.
  • No backend data model exists because the current use case does not require one.
  • No large abstraction layer is introduced until duplication becomes real.

Quality bar

Baseline checks

Even a small site should show basic engineering discipline.

  • Typed page data and route metadata.
  • Linting and formatting as part of routine maintenance.
  • Tests around routing, rendering, and template behavior.

What I would extend next

There is room to make the quality story more visible over time.

  • Add a clearer CI summary.
  • Add accessibility checks and a small end-to-end smoke suite.
  • Add a lightweight performance and bundle budget view.

What is intentionally omitted

Constraints kept on purpose

Some things are not here because they would not yet improve the product enough to justify their cost.

  • No CMS or content service.
  • No authentication or personalization.
  • No observability stack beyond what a simple site needs today.

How this architecture would evolve

If the site grew in scope

The current structure is intended to support the next steps without needing a full rewrite.

  • Content could move to a CMS or API while preserving the route and rendering model.
  • A more explicit backend contract could support richer data-driven pages.
  • Caching, observability, and stronger deployment checks could be layered in gradually.

Shared route registry in practice

Route registry excerpt

This is the central pattern: one explicit route definition carries path, rendering, metadata, navigation, and prerender intent together.

  • The same registry feeds header navigation.
  • The same registry resolves route metadata for SSR and client navigation.
  • The same registry determines which localized pages are prerendered for static output.
